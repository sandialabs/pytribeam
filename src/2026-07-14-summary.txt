pytribeam _version.py release-process fix - summary
2026-07-14

PROBLEM
-------
v0.1.3 was published to PyPI with the correct version, but the zip download
of the tagged release (and any offline install done from that zip via
pytribeam_install.bat) showed pytribeam.__version__ == "0.1.2.dev0" instead
of "0.1.3".

ROOT CAUSE
----------
src/pytribeam/_version.py is tracked in git as a static fallback. hatch-vcs
normally computes the version from git tags at build time (this always works
correctly for the wheel/sdist built in CI). But when git metadata isn't
available - a zip download or an offline --no-build-isolation install with no
.git directory - hatch-vcs falls back to reading whatever is already written
in the tracked _version.py file instead of erroring out.

An earlier fix (7/9/26) removed the release.yml job that used to commit a
regenerated _version.py back to main/dev after tagging, because that commit
landed *past* the tag (poisoning the next dev build's version with .devN) and
because it required pushing directly to protected main. Removing that job was
correct, but it also meant nothing ever updated the tracked _version.py file
again - it just sat stale at whatever value was last hand-edited (0.1.2.dev0),
regardless of what got tagged and published afterward.

DECISION
--------
- Leave the already-published v0.1.3 tag/release as-is (documented, known
  issue for that release's zip download only). Do not force-move the tag.
- Fix applies starting with v0.1.4 and onward.
- Bump _version.py via a GitHub Actions workflow_dispatch job that opens a PR
  (not a purely local script), but keep human review/merge and the actual
  tag push manual - matches the existing "Chad runs the git writes" runbook
  style. No auto-merge, no auto-tag.
- Drop the __git_commit__ field entirely rather than write a value that's
  guaranteed to go stale (a file can't know its own commit hash before it's
  committed). The tag is already the authoritative commit pointer:
  git rev-list -n1 vX.Y.Z

WHAT CHANGED
------------
1. New workflow: .github/workflows/prepare-release.yml
   - workflow_dispatch inputs: version (e.g. "0.1.4", "0.1.4rc1"), branch
     (default "dev").
   - Computes the version tuple the same way hatch-vcs would (via
     packaging.version.Version) and writes src/pytribeam/_version.py in the
     same shape hatch-vcs generates.
   - __commit_id__ stays None (matches what hatch-vcs itself always produces
     in this project's config). __git_commit__ line removed, replaced with a
     comment pointing at the git tag.
   - Opens a PR with the change via peter-evans/create-pull-request. No
     auto-merge - maintainer reviews and merges normally.

2. .github/workflows/release.yml - validate_tag job
   - Added a new step that reads _version.py at the tagged commit, extracts
     __version__, and FAILS the release if it doesn't exactly match the tag
     (github.ref_name stripped of leading "v"). This makes the drift bug
     impossible to ship again - a tag pushed before its bump PR is merged now
     gets rejected with an actionable error, instead of silently shipping a
     correct wheel alongside a stale zip.
   - Updated the github-release job's header comment to reference the new
     prepare-release.yml pre-tag step (previously it only explained why there
     is no commit-back job, without saying what does keep the file current).

3. src/pytribeam/_version.py (the currently tracked file)
   - Removed the __git_commit__ = "3b3dd19" line, replaced with a comment.
     Verified safe: _package_metadata.get_pytribeam_commit_id() reads it via
     getattr(..., None) as the last of three fallbacks, and all of its tests
     already use monkeypatch.setattr(..., raising=False), i.e. absence was
     already a supported/expected outcome. Full test suite (20/20 in
     tests/test__package_metadata.py) still passes after removal.

4. .gitignore
   - Fixed a comment that incorrectly claimed _version.py "is regenerated by
     hatch-vcs on each tagged release (via release.yml)" - that's false,
     release.yml never writes it. Corrected to describe the real process
     (bumped before tagging via prepare-release.yml + a reviewed PR).

5. CONTRIBUTING.md - brought fully in line with the above, specifically:
   - Listed prepare-release.yml in the workflow overview.
   - Clarified the difference between "wheel/sdist version" (always correct,
     computed by hatch-vcs in CI, no manual step) vs. "tracked _version.py in
     the repo" (static fallback, goes stale unless bumped manually - this is
     what broke v0.1.3).
   - Added the bump step as an explicit numbered step in both the rc and the
     final-release walkthroughs (previously only mentioned once, generically,
     disconnected from the actual sequence).
   - Fixed the "Release procedure (quick reference)" section at the bottom,
     which omitted the bump step entirely - following it as originally
     written would have reproduced the v0.1.3 bug.
   - Fixed `python preflight.py --release`, which referenced a flag that does
     not exist in preflight.py (only --no-sync, --skip-network-check,
     --force, --all-tests exist) and falsely claimed it checks branch/push
     state. Replaced with --all-tests (full lint + test run) and pointed
     readers to the actual branch/push-race-condition section instead.
   - Cross-linked all _version.py mentions throughout the doc back to one
     canonical "Bump src/pytribeam/_version.py" section.

6. Memory doc updated (project_release_process, not part of the repo) with
   the same root cause / fix summary for future sessions.

RELEASE PROCESS GOING FORWARD
------------------------------
RC release (TestPyPI, from dev):
  1. gh workflow run prepare-release.yml -f version=0.1.4rc1 -f branch=dev
  2. Review and merge the PR it opens into dev.
  3. git checkout dev && git pull && git push origin dev
  4. git tag -a v0.1.4rc1 -m "Release candidate 1 for version 0.1.4"
     git push origin v0.1.4rc1
  5. release.yml runs: validate_tag (now incl. _version.py-matches-tag check)
     -> test -> build -> GitHub pre-release -> publish to TestPyPI.

Final release (PyPI, from main):
  1. gh workflow run prepare-release.yml -f version=0.1.4 -f branch=dev
     (needed again - the tag string almost always changes from the rc, e.g.
     0.1.4rc1 -> 0.1.4). Merge that PR into dev.
  2. Open and merge a PR from dev into main (merge commit, not squash - main
     is protected).
  3. git fetch origin && git checkout main && git reset --hard origin/main
     git tag -a v0.1.4 -m "Release version 0.1.4"
     git log -1 --oneline v0.1.4^{commit}   # verify before pushing
     git push origin v0.1.4
  4. Same pipeline runs, but produces a full GitHub Release and publishes to
     production PyPI (pausing for manual approval if that gate is
     configured).
